## Planning horizon
Initial horizon: six controlled passes after V1 planning is complete.

## Milestones
### M1. Foundation and positioning
Goal:
- establish scope, narrative, success criteria, and project inventory rules

Deliverables:
- program charter
- roadmap
- decision log
- positioning baseline
- project inventory
- featured-project shortlist

Exit criteria:
- target audience defined
- positioning statement approved
- featured-project direction approved

### M2. Content system
Goal:
- create reusable content and proof structure for all channels

Deliverables:
- master profile
- naming normalization
- project families
- GitHub and LinkedIn plans
- CV strategy

Exit criteria:
- one coherent narrative exists across site, CV, GitHub, and LinkedIn
- flagship project descriptions can be expanded without reopening strategy

### M3. Website adaptation
Goal:
- convert the `salimov` site into a premium scroll-native portfolio

Deliverables:
- revised homepage structure
- project gallery replacement plan
- skills visual spec
- content-to-code mapping in later pass

Exit criteria:
- site structure is vertically native
- project proof system is clear
- legacy template constraints have an explicit removal plan

### M4. Profile and CV alignment
Goal:
- align all external candidate-facing surfaces

Deliverables:
- final CV variant set
- LinkedIn copy
- GitHub profile README plan
- pinned repository priorities

Exit criteria:
- project names and hierarchy match across all channels
- role targeting is consistent

### M5. QA and launch readiness
Goal:
- validate credibility, consistency, UX, and technical readiness

Deliverables:
- QA checklist
- review log in later pass
- launch-readiness pass in later phase

Exit criteria:
- no broken trust signals
- no conflicting claims
- no trapped navigation or inaccessible content

### Pass 05. Visual and proof assets

### Pass 06. QA and launch
Focus:
- proofread

## Prioritization rules
- Fix positioning before visuals
- Fix project selection before long descriptions
- Implement the site only after the content backbone is stable
- Treat GitHub and LinkedIn as support surfaces for the main narrative, not separate brand experiments

## Dependencies
- Website implementation depends on approved positioning and featured-project list
- CV and LinkedIn finalization depend on the same source content used by the website
- GitHub curation depends on project inventory review and repo-quality checks

## Near-term success metrics
- Hero message explains role and value in under 10 seconds
- Featured projects reduced to a focused, defensible shortlist
- Resume, LinkedIn, GitHub, and website share the same target-role framing
- The portfolio can support real applications without custom rewriting each time
