Feeds into: Website, CV, GitHub, LinkedIn consistency QA

## Canonical naming rules
- `Sahara` is the primary public name
- `MyOwnCursor` is a secondary continuity label for the same product
- public names must match across website, CV, GitHub README, and LinkedIn
- `CheckAuto TN`, `DevisPilot`, `TrustKit`, `PricePilot`, `DentoFlow`, `Local-Arcads`, `FlashForge`, `Context Assistant`, `SignLik`, and `CogniCare` should not be renamed casually on different surfaces

## Canonical hierarchy rules
- flagships are the strongest public anchors
- directed products are visible but explicitly in development
- support projects reinforce capability
- internal-only projects should not leak into public copy accidentally

## Surface hierarchy defaults
- Website:
  - flagship
  - directed products
  - top support
- CV:
  - flagship first
  - limited builder + director support
  - very selective directed-product mention
- GitHub:
  - flagship and top-support repos pin first
  - directed products can appear in README or secondary pins if public-safe
- LinkedIn:
  - flagship in Featured
  - directed products in About/Experience only when framed as in development

## CTA consistency
- `View code` only for public-safe and link-ready repos
- `Case study` for flagship and selected directed products
- `Request walkthrough` for sensitive or non-public projects
- `View project` only when there is a meaningful public page or live demo

## Cross-surface checks
- no project appears under two different names without explanation
- no project appears as flagship on one surface and hidden on another
- no directed product is framed as fully launched if the proof-bank says otherwise
- no support project should outrank a flagship in visual emphasis without an intentional reason

## Final pre-launch checklist
- compare website project names against CV bullets
- compare CV names against LinkedIn Featured and Experience
- compare GitHub pinned repos against flagship/top-support hierarchy
- compare all public CTAs against the proof-bank before launch
